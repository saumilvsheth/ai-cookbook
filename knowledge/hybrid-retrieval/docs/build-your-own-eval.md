# Build a BEIR-style eval set from your own corpus

The tutorial evaluates on FiQA because it ships with human-labeled qrels. Your own corpus will not. This doc shows the cheapest path to get a usable eval set anyway: **let an LLM generate the queries against your real documents**, then plug the result straight into [`6-evaluate.py`](../6-evaluate.py).

The output is a folder that looks exactly like FiQA's: three parquet files (`corpus.parquet`, `queries.parquet`, `qrels.parquet`) with the BEIR schema. The rest of the pipeline does not change.

## The intuition

You already have one of the three files: your **corpus**. The trick is to invert the retrieval task. Instead of "given a query, find the relevant doc," ask the LLM "given this doc, what query would a real user ask that this doc would answer?" The doc you fed in becomes a labeled relevant document for the generated query, for free.

That gives you (query, relevant_doc) pairs without any human labelers. Good enough to compare retrieval methods against each other on *your data*, which is what you actually care about.

## The three files you need

| File              | Schema                                  | Where it comes from                                |
| ----------------- | --------------------------------------- | -------------------------------------------------- |
| `corpus.parquet`  | columns: `_id`, `title`, `text`         | Your real documents.                               |
| `queries.parquet` | columns: `_id`, `text`                  | Generated by an LLM, one query per sampled doc.    |
| `qrels.parquet`   | columns: `query-id`, `corpus-id`, `score` | The source doc gets score 1 against its generated query. |

## The recipe in five steps

1. **Sample 100 documents** from your corpus (random, stratified by length, whatever fits your data). 100 is enough to see meaningful NDCG differences between methods, small enough to keep LLM costs negligible.

2. **For each sampled doc, prompt an LLM to generate one realistic user query** the doc would answer. Quality of this prompt matters. The single biggest mistake is generating queries that just paraphrase the doc's first sentence, which makes BM25 look artificially good. Use something like:

   ```
   You are a user with a question. The following document contains the answer.
   Generate a single realistic question that a user would ask, in their own words.
   Rules:
   - The question must be answerable by the document.
   - Use natural language, not the document's exact phrasing.
   - Avoid copying long phrases directly from the document.
   - Keep it under 20 words.

   Document:
   {doc_text}

   Question:
   ```

   Run this in a batch with `gpt-4o-mini` or similar. Save each `(query_id, query_text, source_doc_id)` triple.

3. **Write the three parquet files** in BEIR schema. The qrels file is the simplest: each row is `(query_id, source_doc_id, 1)`.

4. **Optional but recommended: enrich qrels with LLM-as-judge.** For each generated query, run your retrieval pipeline over the corpus, take the top 20 candidates, and ask an LLM to label each one as relevant (1) or not (0). This catches the case where multiple docs in your corpus answer the same question, which your initial single-source-doc qrels miss. Without this step, your NDCG numbers are pessimistic (a perfectly-correct retrieval that returns a *different* answering doc would score 0 because that doc is not in your qrels). With it, your numbers more closely match what a human would say.

5. **Point `6-evaluate.py` at your data.** Either drop your parquets into `knowledge/hybrid-retrieval/data/fiqa/` (overwrites the FiQA data) or update `DATA_DIR` in [`utils/retrievers.py`](../utils/retrievers.py) to point at your folder. Re-run the embed and BM25 index scripts on your new corpus, then `6-evaluate.py` prints NDCG@10 for all four methods on your data.

## Why this works for comparison even if absolute numbers drift

The eval set is biased by the LLM that generated the queries. The absolute NDCG@10 numbers are not directly comparable to BEIR leaderboard scores. **But the relative ordering between methods on *your data* is still meaningful.** If `Hybrid + Rerank` lifts NDCG@10 from 0.38 to 0.51 on your synthetic eval, that lift will almost certainly translate to your real users too. That's the feedback loop you actually need.

## Practical sizing

| Scale                 | Queries to generate | Cost (gpt-4o-mini @ $0.15/1M input, $0.60/1M output) |
| --------------------- | ------------------- | ---------------------------------------------------- |
| Quick sanity check    | 50                  | <$0.01                                               |
| Useful for comparison | 100-200             | ~$0.05                                               |
| Confidence intervals  | 500-1000            | ~$0.25                                               |

If you want to be rigorous, generate the eval set once and freeze it. Re-running with the LLM regenerates different queries and makes runs non-comparable.
